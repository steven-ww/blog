Java after January 2019

Java Azure

There have been a number of blogs and podcasts discussing Oracles new Java release schedule and support model, but I thought I’d try describe the concerns and options here in my own way and include my view of the latest changes (as of this writing).

Release Cadence.

Oracle is releasing a new version of Java every 6 months, with a long term support version every 3 years. Examples of this are

Java 9, September 2017

Java 10, March 2018

Java 11, September 2018 (LTS)

Java 12, March 2019

The LTS version means that Oracle will continue to provide patches and security fixes to this version for a number of years. The exact number of years depends on the support contract you purchase from Oracle.

Support/Access to Security and Bug Fixes 

There are some items that we should unpack with regard to the above. Firstly, an LTS version doesn’t mean you’ll get Long Term Support. What it does mean is that you’ll get long term support if you purchase a support contract. What I’ve heard promoted a lot with regard to this is that in order to stay in a supported version (one for which you can get security fixes and patches) without a support contract, you’d need to upgrade to the new version of Java every 6 months.

This presupposes that Oracle provides public access to fixes for each Java release for 6 months. You’d use the latest version of Java for six months while the public support is available, then move to the next version when it’s released and get 6 months of public fixes on that version.

One issue with the above approach is the practicality of it. Just how reasonable is it so assume you can successfully migrate and test all your applications on the new version of Java every six months?

Another issue, and one I haven’t seen mentioned at all, is Oracles new license terms, starting with Java 11. This prevents any commercial use of Java 11. So while you will have access to Java 11 public releases for 6 months, you may not use them in production.

An extract from the new Oracle Java License agreement :

“License Rights and Restrictions
Oracle grants You a nonexclusive, nontransferable, limited license to internally use the Programs, subject to the restrictions stated in this Agreement and Program Documentation, only for the purpose of developing, testing, prototyping and demonstrating Your Application and not for any other purpose. “

https://www.oracle.com/technetwork/java/javase/terms/license/javase-license.html

In my mind this completely invalidates any attempt to use Oracles Java in a prod environment without first purchasing a support agreement, even if you do intend to upgrade to the latest Java every 6 months.

Okay, what about Java 8?

Java 8 currently still has patches and security fixes made available publicly. So, for the moment you can continue to use Java 8 and download the latest updates without a support agreement.

As of January next year though (2019), this falls away. While the updates will most likely still be available for download after January, you may not use those in a production environment without a support contract with Oracle.

All that means?

For Production/Commercial use, without a support contract:

  • To use Java 8 after January is okay, if you don’t upgrade/download any patch releases after January. (You’ll have no support for fixing security vulnerabilities and bugs.)
  • Using Java 11 is not possible.
  • Java 12…? Depends what license applies… If it’s the Java 11 license, then no, not an option.

Alternatives?

OpenJDK is, as of Java 11, virtually identical to the Oracle Java 11. Oracle have either bundled the previously commercial tools supplied with Oracle Java into OpenJDK or removed them from both the OpenJDK and Oracle JDK. This applies to other items to, like WebStart (removed) and JavaFX (moved to an open source project OpenJFX). Oracle even pushes changes/fixes through OpenJDK before bundling them into Oracles builds of the JDK.

OpenJDK however needs you to build the binaries. Not something everybody is setup to do. Thankfully a number of companies are providing binaries of the OpenJDK.

Zulu from Azul systems provide tested and certified builds, available for commercial use and you can purchase support.  https://www.azul.com/downloads/zulu/

AdoptOpenJDK provides binaries of Java 8 and Java 11. https://adoptopenjdk.net

Security Fixes/Patches for OpenJDK?

Okay, so we’ve moved away from Oracle Java to OpenJDK. We have a binary package we can consume. Now though, do we have ongoing support for bug and security fixes?

Oracle won’t apply security and bug fixes to the OpenJDK after the public support period for a release ends. That means that after 6 months, Oracle will not apply any fixes it may release in its commercial offering to the OpenJDK project.

Vendors like Azul, will apply these fixes back to their binary distributions, but again, possibly only to customers who have a support contract with them.

The open source community for OpenJDK will, most likely apply patches and security fixes to OpenJDK LTS versions (so beyond the 6 months). This will however be at the pace and priority set by the community.

My hope is that OpenJDK will become the mainstream Java version and will receive the attention it needs to ensure it has a long term support model that delivers patches and security fixes in a manner that supports its commercial use with no more risk than we currently have with the Oracle releases.
